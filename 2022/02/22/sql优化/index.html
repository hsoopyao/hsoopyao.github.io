🐯sql优化

本文最后更新于：2022年2月24日 下午

MySQL

三大范式

1NF：数据库表的每一列都是不可分割的原子数据项
2NF：数据库表中的每一列都和主键相关，而不能只与主键的某一部分相关，即不存在部分依赖(一张表描述一件事情)
3NF：保证每列都和主键直接相关

sql执行过程

sql语句 → 进入缓存查找 → 解析器(查找语法是否有错) → 预处理 → 查询优化器 → 执行方案 → 查找引擎 → 引擎执行查找 → 数据

其中优化的点是 ==查询优化器 → 执行方案==

sql执行顺序

  1. FROM 选择表
  2. ON 连接条件
  3. JOIN 连接表
  4. WHERE 过滤条件
  5. GROUP BY 分组
  6. HAVING 分组过滤
  7. SELECT 选择字段
  8. DISTINCT 去重
  9. ORDER BY 排序
  10. LIMIT 分页

InnoDB与MyISAM的区别

  • InnoDB支持外键、事务，支持行级锁，会缓存索引与数据
  • MyISAM不支持事务，支持表级锁，只缓存索引

索引

❓什么是索引
索引是一种数据结构，能够帮助我们快速的检索数据库中的数据

❓索引具体采用哪种数据结构
常见的MySQL主要有两种结构：HASH索引和B+树索引；InnoDB引擎默认是B+树

HASH索引

优点：

  1. Hash索引可以根据hash值直接定位到数据的存储位置，适用于等值查询
    缺点：
  2. 不适用于查询排序，因为hash后的数据不会像原有数据一样保持顺序
  3. 不适用于模糊查询
  4. 不支持排序，也不支持范围查询
  5. 不支持多列联合索引的最左匹配规则
  6. 如果有大量重复键值得情况下，哈希索引的效率会很低，因为存在哈希碰撞问题

B+Tree索引

🔖B+tree是一种多路平衡查询树，它的节点是天然有序的(左节点小于父节点，右节点大于父节点)

特点：

  1. B+Tree根节点和支节点没有数据区，关键字对应的数据只保存在叶子节点中。即只有叶子节点的关键字数据区才会保存真正的数据内容和内容的地址
  2. B+Tree中的叶子节点不会去保存子节点的引用
  3. B+Tree叶子节点是顺序排列的，并且相邻的节点具有顺序引用的关系

聚簇索引、非聚簇索引

❓什么是聚簇索引
B+Tree中的叶子节点存储的是当前行的数据的是主键索引，不需要再次回表进行查询

❓什么是非聚簇索引
B+Tree中的叶子节点存储的是主键的值是非主键索引，需要再次回表查询

❓什么是回表
先通过普通索引的值定位聚簇索引值，再通过聚簇索引的值定位行记录数据，需要扫描两次索引B+树，它的性能较扫一遍索引树更低。

面试题：

主键索引查询只会查一次，而非主键索引需要回表查询多次，是所有情况都是这样的吗？非主键索引一定会查询多次吗？

覆盖索引（covering index） 指一个查询语句的执行只用从索引中就能够取得，不必从数据表中读取。也可以称之为实现了索引覆盖。 当一条查询语句符合覆盖索引条件时，MySQL只需要通过索引就可以返回查询所需要的数据，这样避免了查到索引后再返回表操作，减少I/O提高效率。 如，表covering_index_sample中有一个普通索引 idx_key1_key2(key1,key2)。当我们通过SQL语句：select key2 from covering_index_sample where key1 = ‘keytest’;的时候，就可以通过覆盖索引查询，无需回表。

覆盖索引

❓什么是覆盖索引
覆盖索引（covering index ，或称为索引覆盖）即从非主键索引中就能查到的记录，而不需要查询主键索引中的记录，避免了回表的产生减少了树的搜索次数，显著提升性能。

每一个索引在InnoDB中对应着一颗B+Tree，主键索引的树叶子节点中存储的是整行数据而非主键索引的树叶子节点存储的是主键id。

  • 当我们在name字段上建立普通索引，在不使用覆盖索引的情况下
    select age from student where name = '小李';

    流程为：
  1. 通过name索引树拿到名称为小李的id
  2. 通过id索引树找到对应id的学生信息
  3. 从信息中筛选出age字段返回
  • 当我们引入覆盖索引，以name和age两个字段建立联合索引
    create index cover_index_name on student(name,age);

select age from student where name = '小李';

    流程为：
  1. 通过name,age的联合索引树中找到小李的节点
  2. 此时节点索引中包含age属性，直接返回

联合索引

❓什么是联合索引
两个或者更多列上的索引被称为联合索引，联合索引又称复合索引。对于复合索引:MySQL从左到右使用索引中的字段，一个查询可以只使用索引中的一部分，但只能是最左侧部分。例如索引是 key index(a,b,c) ，可以支持 a|a,b|a,b,c 三种组合进行查找，但不支持 b,c 进行查找(最左匹配原则)，当最左侧字段是常量引用时，联合索引就十分有效。

索引失效情况：

  1. where条件中使用or
  2. 一般情况下不鼓励使用like操作，如果非使用不可，如何使用也是一个问题。like “%aaa%” 不会使用索引，而like “aaa%”可以使用索引。
  3. NOT IN操作不会使用索引将进行全表扫描。NOT IN可以用NOT EXISTS代替

全文索引

❓什么是全文索引
全文索引,通过建立倒排索引,可以极大的提升检索效率,解决判断字段是否包含的问题

版本支持：

  1. MySQL5.6之前的版本，只有MyISAM引擎支持全文索引
  2. MySQL5.6后的版本，MyISAM和InnoDB引擎都支持全文索引
  3. 只有字段是char,varchar,text及其系列才可以支持全文索引
  4. 在MySQL5.7.6版本中，内置了ngram全文解析器，用来支持亚洲语种的分词
  • 在学生表中添加全文索引
    create fulltext index full_index_name on student(sname) WITH PARSER ngram;
  • 使用全文索引
    select * from student where match(sname) against ('test_1');

注意：match()中的字段必须与定义全文索引时的字段保持一致，否则会报错，无法使用全文索引

SQL优化

  1. 尽量不要使用select * ，只取有需要的字段
  2. or会导致索引失效，可以用union代替
  3. like属于范围查询，以通配符开头时不使用索引，最好通配符在右边，采用覆盖索引可以解决like%xxx%时索引失效
  4. 复合索引，最左匹配原则，把常用的条件放在最左边
  5. 覆盖索引，只通过索引就可以查询需要的数据，查询的列，要在所建的索引的范围中
  6. 在索引字段上使用内置函数会导致索引失效
  7. 避免在where中对字段进行表达式操作（SQL解析时，如果字段相关的是表达式就进行全表扫描）
  8. 避免在where子句中使用!= 或者 <>，这会让索引失效
  9. varchar,字符串必须有单引号，有可能会导致类型转换导致索引失效，有可能会导致行级锁升级为表级锁使得阻塞(因为行级锁是基于索引的，用不到索引不会使用行级锁)

in 和 exist 的选择

  • 当A表大于B表数据时，使用in b>a时使用exist
  • in是先执行in内部的语句
  • exist是先执行外部语句
  • 选择就是看两张表的数据量进行选择

🔖索引下推
select * from table where username = 'zhangsan' and age = 20;

  • 在没有索引下推之前，先根据username从存储引擎中拉取数据到server层，然后在server层中对age进行数据过滤
  • 有索引下推之后，根据name和age两个条件来做数据筛选，将筛选后的结果返回给server层
  • 这样返回给MySQL服务端的索引数又会减少。有了索引下推优化，可以在有like条件查询的情况下，减少回表次数。

面试题：那什么情况下会发生明明创建了索引，但是执行的时候并没有通过索引呢？

查询优化器 一条SQL语句的查询，可以有不同的执行方案，至于最终选择哪种方案，需要通过优化器进行选择，选择执行成本最低的方案。 在一条单表查询语句真正执行之前，MySQL的查询优化器会找出执行该语句所有可能使用的方案，对比之后找出成本最低的方案。这个成本最低的方案就是所谓的执行计划。 优化过程大致如下：

  1. 根据搜索条件，找出所有可能使用的索引
  2. 计算全表扫描的代价
  3. 计算使用不同索引执行查询的代价
  4. 对比各种执行方案的代价，找出成本最低的那一个

🔖type
显示的是访问类型，是较为重要的一个指标

type 说明

system 这是const连接类型的一种特例，表仅有一行满足条件。
NULL 意味说mysql能在优化阶段分解查询语句，在执行阶段甚至用不到访问表或索引（高效）

🔖Extra
输出中另外一个很重要的列，该列显示MySQL在查询过程中的一些详细信息，MySQL查询优化器执行查询的过程中对查询计划的重要补充信息。

类型 说明
Using filesort MySQL有两种方式可以生成有序的结果，通过排序操作或者使用索引，当Extra中出现了Using filesort 说明MySQL使用了后者，但注意虽然叫filesort但并不是说明就是用了文件来进行排序，只要可能排序都是在内存里完成的。大部分情况下利用索引排序更快，所以一般这时也要考虑优化查询了。使用文件完成排序操作，这是可能是ordery by，group by语句的结果，这可能是一个CPU密集型的过程，可以通过选择合适的索引来改进性能，用索引来为查询结果排序。
Using temporary 用临时表保存中间结果，常用于GROUP BY 和 ORDER BY 操作中，一般看到它说明查询需要优化了，就算避免不了临时表的使用也要尽量避免硬盘临时表的使用
Not exists MySQL优化了LEFT JOIN，一旦它找到了匹配LEFT JOIN标准的行，就不再搜索了
Using index 说明查询是覆盖了索引的，不需要读取数据文件，从索引树（索引文件）中即可获得信息。如果同时出现using where，表明索引被用来执行索引键值的查找，没有using where，表明索引用来读取数据而非执行查找动作。这是MySQL服务层完成的，但无需再回表查询记录。
Using index condition 这是MySQL 5.6出来的新特性，叫做“索引条件推送”。简单说一点就是MySQL原来在索引上是不能执行如like这样的操作的，但是现在可以了，这样减少了不必要的IO操作，但是只能用在二级索引上。
Using where 使用了WHERE从句来限制哪些行将与下一张表匹配或者是返回给用户。注意：Extra列出现Using where表示MySQL服务器将存储引擎返回服务层以后再应用WHERE条件过滤。
Using join buffer 使用了连接缓存：Block Nested Loop，连接算法是块嵌套循环连接;Batched Key Access，连接算法是批量索引连接
impossible where where子句的值总是false，不能用来获取任何元组
select tables optimized away 在没有GROUP BY子句的情况下，基于索引优化MIN/MAX操作，或者对于MyISAM存储引擎优化COUNT( * )操作，不必等到执行阶段再进行计算，查询执行计划生成的阶段即完成优化。
distinct 优化distinct操作，在找到第一匹配的元组后即停止找同样值的动作

锁

InnoDB实现了以下两种类型的行锁：

  • 共享锁(S)(读锁)：所有事务只能读(可以拿到读锁)但不能修改
    • select * from table where ... lock in share mode;
  • 排他锁(X)(写锁)：只能获得锁才能进行操作
    • select * from table where ... for update

加锁原则：

  1. 加锁的基本单位是next-key
  2. 查找过程中访问到的对象才会加锁

优化：

  1. 索引上的等值查询，给唯一索引加锁的时候，next-key退化为间隙锁
  2. 索引上的等值查询，向右遍历时且最后一个值不满足等值条件的时候，next-key退化为间隙锁

行锁

每一个InnoDB表都需要一个聚簇索引，有且只有一个。如果你为该表表定义一个主键，那么MySQL将使用主键作为聚簇索引；如果你不为定义一个主键，那么MySQL将会把第一个唯一索引（而且要求NOT NULL）作为聚簇索引；如果上诉两种情况都GG，那么MySQL将自动创建一个名字为GEN_CLUST_INDEX的隐藏聚簇索引。

因为是聚簇索引，所以B+树上的叶子节点都存储了数据行，那么如果现在是二级索引呢？InnoDB中的二级索引的叶节点存储的是主键值（或者说聚簇索引的值），所以通过二级索引查询数据时，还需要将对应的主键去聚簇索引中再次进行查询。

关于索引的问题就到这，我们用一张直观的图来表示行锁：

接下来以两条SQL的执行为例，讲解一下InnoDB对于单行数据的加锁原理：

update user set age = 10 where id = 49;
update user set age = 19 where name = 'Tom';

第一条SQL使用主键查询，只需要在id = 49这个主键索引上加锁。
第二条SQL使用二级索引，那么首先在name = ‘Tom’这个索引上加写锁，然后由于使用InnoDB二级索引还要再次根据主键索引查询，所以还要在id = 49这个主键索引上加锁
也就是说使用主键索引需要加一把锁，使用二级索引需要在二级索引和主键索引上各加一把锁

根据索引对单行数据进行更新的加锁原理了解了，那如果更新操作涉及多个行呢，比如下面 SQL 的执行场景。

update user set age = 10 where id > 49;

上述SQL执行过程如下图所示，MySQL Server会根据WHERE条件根据第一条满足条件的记录，然后InnoDB引擎会将第一条记录返回并加锁，接着MySQL Server发起更新行记录的UPDATE请求，更新这条记录。一条记录操作完成，再读取下一条记录，直至没有匹配的记录为止

行锁的类型

不同的锁锁定的位置是不同的，比如说记录锁只锁定对应的记录，而间隙锁锁住记录和记录之间的间隙，Next-key Lock则锁住所属记录之间的间隙。不同的锁类型锁定的范围大致如图所示：

🔖记录锁(Record Lock)
锁定某行记录，行锁是加在索引上的，==如果当你的查询语句不走索引的话，那么它就会升级到表锁==，最终造成效率低下，所以在写SQL语句时需要特别注意。

🔖间隙锁(Gap Lock)
当我们使用范围条件而不是相等条件去检索，并请求锁时，InnoDB就会给符合条件的记录的索引项加上锁；而对于键值在条件范围内但并不存在（参考上面所说的空闲块）的记录，就叫做间隙，InnoDB在此时也会对间隙加锁，这种记录锁+间隙锁的机制叫Next-Key Lock

间隙锁的目的是为了防止幻读，其主要通过两个方面实现这个目的：

  1. 防止间隙数据内有新数据插入
  2. 防止已存在的数据，更新成间隙内的数据

InnoDB自动使用间隙锁的条件：

  1. 必须在RR级别下
  2. 检索条件必须有索引(没有索引的话，MySQL会扫描全表，那样会锁定整张表所有的记录，包括不存在的记录，此时事务不能修改不能删除不能添加)

🔖临键锁(Next-key Lock)
临键锁是记录锁与与间隙锁的结合，所以临键锁与间隙锁是一个同时存在的概念，并且临键锁是个左开有闭的却比如(16, 18]。

事务

❓什么是事务
数据库中的事务是指对数据库执行一批操作，在同一个事务当中，这些操作最终要么全部执行成功，要么全部失败，不会存在部分成功的情况。

事务的四大特性(ACID)

  1. 原子性：要么全部成功，要么全部失败
  2. 一致性：在一个事务内，数据是一致的
  3. 持久性：当操作持久化后，数据无法回滚
  4. 隔离性：事务与事务之间的操作是隔离的，互不干扰的

事务问题与隔离级别

问题概念

🔖更新丢失
两个不同事务在某一时刻对同一组数据进行读取后，先后进行修改，导致第一次操作数据丢失

🔖脏读
一个事务执行的过程中读取到了其他事务未提交的数据

🔖不可重复读
在同一事务中，多次读取同一数据返回的结果有所不同

🔖幻读
事务A在操作一堆数据的时候，事务B插入了一条数据，A事务再次（第二次）查询，发现多了一条数据，像是幻觉。与不可重复读类似，不同的是一个是修改删除操作，一个是新增操作。

隔离级别

事务隔离级别主要是解决了上面多个事务之间数据可见性及数据正确性的问题。

  1. 读未提交：READ-UNCOMMITTED
  2. 读已提交：READ-COMMITTED
  3. 可重复读：REPEATABLE-READ
  4. 串行：SERIALIZABLE（多个事务之间读写、写读、写写会产生互斥，效果就是串行执行，多个事务之间的读读不会产生互斥）

各种隔离级别中会出现的问题

隔离级别 脏读 不可重复读 幻读
READ-UNCOMMITTED 有 有 有
READ-COMMITTED 无 有 有
REPEATABLE-READ 无 无 有
SERIALIZABLE 无 无 无

虽然串行不会有脏读、不可重复读、幻读的问题，但是效率会大大下降，应用程序的性能会急剧降低。

Variable_name Value

MVCC

❓什么是MVCC
全称Multi-Version Concurrency Control，多版本并发控制。MVCC 是一种并发控制的方法，一般在数据库管理系统中，实现对数据库的并发访问。

当前读

❓什么是当前读
读取的数据库记录，都是当前最新的版本，会对当前读取的数据进行加锁，防止其他事务修改数据，是悲观锁的一种操作

如下操作都是当前读：

-- 共享锁
select ... lock in share mode;
-- 排他锁
select ... for update;
-- 排他锁
update ...;
-- 排他锁
insert ...;
-- 排他锁
delete ...;
-- 串行化事务隔离级别

快照读的实现是基于多版本并发控制，即MVCC，既然是多版本，那么快照读读到的数据不一定是当前最新的数据，有可能是之前历史版本的数据。

实现原理

主要是以下三样来实现的：

  • 版本链，其中包含三个隐藏字段（db_trx_id，db_roll_pointer，db_row_id）
  • undo日志
  • Read View

版本链 undolog

🔖db_trx_id
创建或者最后一次修改该记录的事务id

🔖db_roll_pointer
回滚指针，指向这条记录的上一个版本（存储于rollback segment里）

🔖db_row_id
隐含的自增主键（隐藏主键），如果数据表没有主键，InnoDB会自动以db_row_id产生一个聚簇索引

每次对数据库记录进行改动，都会记录一条undolog，每条undolog都有一个roll_pointer属性(INSERT操作对应的undolog没有该属性，因为记录并没有历史版本)，可以将这些undo日志都连起来，穿成一个链表。

对该记录每次更新后，都会将旧值放到一条undo日志中，就算是该记录的一个旧版本，随着更新次数的增多，所有的版本都会被roll_pointer属性连接成一个链表，我们把这个链表称之为版本链，版本链的头节点就是当前记录最新的值。另外，每个版本中还包含生成该版本时对应的事务id，这个信息很重要，在根据ReadView判断版本可见性的时候会用到。

Read View

事务进行快照读操作的时候生产的读视图(Read View)，在该事务执行的快照读的那一刻，会生成数据库系统当前的一个快照。

记录并维护系统当前活跃事务的ID(没有commit，当每个事务开启时，都会被分配一个ID, 这个ID是递增的，所以越新的事务，ID值越大)，是系统中当前不应该被本事务看到的其他事务id列表。

Read View主要是用来做可见性判断的, 即当我们某个事务执行快照读的时候，对该记录创建一个Read View读视图，把它比作条件用来判断当前事务能够看到哪个版本的数据，既可能是当前最新的数据，也有可能是该行记录的undo log里面的某个版本的数据。

属性：

  • trx_ids：当前系统活跃（未提交）事务版本号集合
  • low_limit_id：创建当前read view时 当前系统最大事务版本号+1
  • up_limit_id：创建当前read view时 系统正处于活跃事务最小版本号
  • create_trx_id：创建当前read view的事务版本号

read view可见性判断条件：

  • db_trx_id < up_limit_id || db_trx_id == creator_trx_id（显示）
    • 如果数据事务ID小于read view中的最小活跃事务ID，则可以肯定该数据是在当前事务启之前就已经存在了的,所以可以显示。
    • 或者数据的事务ID等于creator_trx_id ，那么说明这个数据就是当前事务自己生成的，自己生成的数据自己当然能看见，所以这种情况下此数据也是可以显示的。
  • db_trx_id >= low_limit_id（不显示）
    • 如果数据事务ID大于read view 中的当前系统的最大事务ID，则说明该数据是在当前read view 创建之后才产生的，所以数据不显示。如果小于则进入下一个判断
  • db_trx_id是否在活跃事务（trx_ids）中
    • 不存在：则说明read view产生的时候事务已经commit了，这种情况数据则可以显示。
    • 已存在：则代表我Read View生成时刻，你这个事务还在活跃，还没有Commit，你修改的数据，我当前事务也是看不见的。
