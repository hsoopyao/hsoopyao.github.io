接口返回统一数据格式

现在是前后端分离的时代，在编写后端开发时要返回统一数据格式(一般都是json字符串)，这样才能更好的查看返回的结果

一般包含：

  • 信息 Message
  • 状态码 Code
  • 结果 Result/Data
  • 时间戳 Timestamp

代码示例 

public class Result<T> implements Serializable {
    
    private static final long serialVersionUID = 1L;
    
     /**
     * 返回处理消息
     */
    @ApiModelProperty(value = "返回处理消息")
    private String message = "操作成功！";

    /**
     * 返回代码
     */
    @ApiModelProperty(value = "返回代码")
    private Integer code = 0;

    /**
     * 返回数据对象 data
     */
    @ApiModelProperty(value = "返回数据对象")
    private T result;

    /**
     * 时间戳
     */
    @ApiModelProperty(value = "时间戳")
    private long timestamp = System.currentTimeMillis();
    
     public static <T> Result<T> OK() {
        Result<T> r = new Result<T>();
        r.setSuccess(true);
        r.setCode(CommonConstant.SC_OK_200);
        r.setMessage("成功");
        return r;
    }

    public static <T> Result<T> OK(T data) {
        Result<T> r = new Result<T>();
        r.setSuccess(true);
        r.setCode(CommonConstant.SC_OK_200);
        r.setResult(data);
        return r;
    }

    public static <T> Result<T> OK(String msg, T data) {
        Result<T> r = new Result<T>();
        r.setSuccess(true);
        r.setCode(CommonConstant.SC_OK_200);
        r.setMessage(msg);
        r.setResult(data);
        return r;
    }

    public static Result<Object> error(String msg) {
        return error(CommonConstant.SC_INTERNAL_SERVER_ERROR_500, msg);
    }

    public static Result<Object> error(int code, String msg) {
        Result<Object> r = new Result<Object>();
        r.setCode(code);
        r.setMessage(msg);
        r.setSuccess(false);
        return r;
    }

    public Result<T> error500(String message) {
        this.message = message;
        this.code = CommonConstant.SC_INTERNAL_SERVER_ERROR_500;
        this.success = false;
        return this;
    }
}

统一捕获全局异常

在方法中捕获异常，使用大量的try…catch会造成代码冗余且不美观

因此可以通过捕获全局异常来整洁代码

使用@RestControllerAdvice和@ExceptionHandler

  • @RestControllerAdvice = @ControllerAdvice+@ResponseBody，@RestControllerAdvice实现全局异常处理，只需要定义类，该类中可以定义多个方法，不同的方法处理不同的异常

  • @ExceptionHandler注解来指明异常的处理对应的类型(value指定捕获的具体异常)

    • 需要注意的是使用@ExceptionHandler注解传入的参数可以一个数组，且使用该注解时，传入的参数不能相同，也就是不能使用两个@ExceptionHandler去处理同一个异常。如果传入参数相同，则初始化ExceptionHandler时会失败。
@RestControllerAdvice
@Slf4j
public class GlobalExceptionAdvice {
    //捕获运行时异常
    @ExceptionHandler(value = RuntimeException.class)
    public Result runtimeExceptionHandler(HttpServletRequest req, RuntimeException e) throws Exception {
        Result result = new Result<>();
        log.error(e.getMessage());
        result.setMsg("系统错误:"+ e.getMessage());
        result.setCode(Status.SYS_ERROR.getCode());
        result.setData("ErrorUrl: "+req.getRequestURL().toString());
        return result;
    }
    
    //也可以捕获自定义异常
    @ExceptionHandler(value = GlobalException.class)
    public Result GlobalExceptionHandler(HttpServletRequest req, GlobalException e) throws Exception {
        Result result = new Result<>();
        result.setMsg(e.getMessage());
        result.setCode(e.getStatus().getCode());
        result.setData("ErrorUrl: "+req.getRequestURL().toString());
        return result;
    }
}

注意：

  • 多个@RestControllerAdvice是有加载顺序的，我们可以在类上添加@Order来控制执行的顺序。
    • 源码分析结果为：@ControllerAdvice或者@RestControllerAdvice上有没有标注@Order注解，或者是@Priority注解，并没有检查有没有实现Ordered接口，故而无效，单单只检查了类上有无这俩注解。

举例：
A类和B类都有@ControllerAdvice注解，要捕获的异常为自定义异常CustomException。
场景一：A中有@ExceptionHandler(CustomException.class)，B中没有，但B中有@ExceptionHandler(Exception.class)，

  • 若B加载顺序优先于A，则CustomException异常会被B处理，因为Exception是CustomException的父类。
  • 若A加载顺序优先于B，则CustomException异常会被A处理。

场景二：A中有@ExceptionHandler(CustomException.class)，B中没有，且B中没有任何@ExceptionHandler()是CustomException的父类

  • 不管AB是何优先级加载，均会被A处理

查询提供自定义排序

案发现场：

实现后进一步建议：

一般分页查询都有两个固定的参数

  • pageSize 页面大小
  • pageNo 当前页数

现在就多加一个字段orderby来控制排序的顺序和要排序的字段

代码示例

public Result<?> list(OpendayLive opendayLive,
                                   @RequestParam(name = "pageNo", defaultValue = "1") Integer pageNo,
                                   @RequestParam(name = "pageSize", defaultValue = "10") Integer pageSize,
                                   @RequestParam(name = "orderby",defaultValue = "") String orderby,
                                   HttpServletRequest req) {
        QueryWrapper<OpendayLive> queryWrapper = QueryGenerator.initQueryWrapper(opendayLive, req.getParameterMap());
        Page<OpendayLive> page = new Page<OpendayLive>(pageNo, pageSize);
    	//判断orderby是否为空
        if (!StrUtil.isBlank(orderby)){
            String[] orderBy = orderby.split(" ");
            String param = orderBy[1];
            if ("asc".equals(param) || "ASC".equals(param)){
                queryWrapper.orderByAsc(orderBy[0]);
            }else {
                queryWrapper.orderByDesc(orderBy[0]);
            }
        }
        IPage<OpendayLive> pageList = opendayLiveService.page(page, queryWrapper);
        return Result.OK(pageList);
    }

问题：

一开始我想着用户的id是唯一且不重复的，所有我用去掉-的uuid作为表的主键

这样导致什么问题呢？

MySql的底层数据结构就是用的B+树，查找是根据表的主键来查找的(主键索引)

如果你的表主键是随机无规律而且非常大的数，如果插入或删除的记录的主键比之前的记录小，需要为新行寻找新的合适的位置从而来分配新的空间，更新树结构，这产生了很多IO操作

如果用自增id来作为表主键，那么树结构则会非常清晰，插入的话直接追加，查找相对也比较快。

冒烟测试

冒烟测试是在软件开发过程中的一种针对软件版本包的快速基本功能验证策略，是对软件基本功能进行确认验证的手段，并非对软件版本包的深入测试。冒烟测试也是针对软件版本包进行详细测试之前的预测试，执行冒烟测试的主要目的是快速验证软件基本功能是否有缺陷。如果冒烟测试的测试例不能通过，则不必做进一步的测试。进行冒烟测试之前需要确定冒烟测试的用例集，对用例集要求覆盖软件的基本功能。这种版本包出包之后的验证方法通常称为软件版本包的门槛用例验证。

在将代码完成后要在本地运行通过方可上传到生产环境上。

贫血模式与充血模式

贫血模式
领域对象中只有get和set方法，或者包含少量的CRUD方法，所有的业务逻辑都不包含在内而是放在Business Logic层

栗子：


public class User {

	private String name;
	
	public String getName(){
		return this.name;
	}
	
	public void setName(String name){
		this.name = name;
	}
	
}

充血模式
领域对象中不仅有get和set方法，还有将大部分业务逻辑和持久化放在Domain Object中，Business Logic只是封装简单的业务逻辑

栗子：


public class User {
	
	private String name;
	
	private Integer status;
	
	public String getAccount(){
		return this.account;
	}
	
	public void setAccount(String account){
		this.account = account;
	}
	
	public Integer getStatus(){
		return this.status;
	}
	
	public void setStatus(String status){
		this.status = status;
	}
	
	//除了get set 方法以外 有别的逻辑处理方法
	public static User create(String input) {
		User entity = new User();
		entity.setName(input);
		entity.setStatus(0);
		return entity;
	}
	
}

Spliter工具使用

概述：
Java中的String.spilit函数会将丢弃部分分隔符

Splitter 可以让你使用一种非常简单流畅的模式来控制这些令人困惑的行为

Splitter.on(',')
    .trimResults()
    .omitEmptyStrings()
    .split("foo,bar,,   qux");

以上代码将会返回 Iterable<String> ，包含 “foo”、 “bar”、 “qux”。一个 Splitter 可以通过这些来进行划分：Pattern、char、 String、CharMatcher。

如果你希望返回的是 List 的话，可以使用这样的代码 Lists.newArrayList(splitter.split(string))。

工厂函数

方法 描述 栗子
Splitter.on(char) 基于特定字符划分 Splitter.on(",")
Splitter.on(String) 基于字符串划分 Splitter.on("11")
Splitter.onPattern(String) 基于正则表达式划分 Splitter.onPattern("\r?\n")

修改器

方法 描述 栗子
omitEmptyStrings() 移除结果的空字符串 Splitter.on(',').omitEmptyStrings().split("a,,c,d") returns "a", "c", "d"
trimResults() 移除结果的空格删除 Splitter.on(',').trimResults().split("a, b, c, d") returns "a", "b", "c", "d"
trimResults(CharMatch) 移除匹配字符 Splitter.on(',').trimResults(CharMatcher.is('_')).split("_a ,_b_ ,c__") 返回 "a ", "b_ ", "c"
limit(int) 达到指定数目后停止字符划分 Splitter.on(',').limit(3).split("a,b,c,d") returns "a", "b", "c,d"

如果想要得到List集合，就使用splitToList()，否则将会得到一个Iterable<String>

Map Splitters
我们可以通过withKeyValueSeparator()指定第二个分割符号，划分为key-value的形式，返回Map<String,String>

@Test  
public void test2(){  
	 Map<String, String> split = Splitter.on("#").withKeyValueSeparator(":").split("1:2#3:4");  
	 System.out.println(split);  
}

返回：{1=2, 3=4}

需要注意的是，`MapSplitter` 对键值对格式有着严格的校验，下例会抛出 `java.lang.IllegalArgumentException` 异常

在服务器上部署项目，mysql容器很容易被入侵，不要映射端口到宿主机，用**ssh forword(端口转发)**的方式访问mysql容器比较安全。

最后通过可视化图形界面连接即可。

如何查看容器ip?

  • 通过docker network ls命令查询所有网络配置
  • 使用docker network inspect 网络配置名称可以查询到网络详情以及每个容器的ip、名称等信息

图片预览与下载

通过访问图片url，会出现两种情况

  1. 下载
  2. 预览

  • 设置content-type:image/png来告诉服务器上传文件的类型；

    • 常见的媒体格式类型有：

      格式 说明
      text/html HTML格式
      text/plain 纯文本格式
      text/xml XML格式
      image/gif GIF图片格式
      image/jpeg JPEG图片格式
      image/png PNG图片格式

    • 以application开头的媒体格式类型：

      格式 说明
      application/xhtml+xml XHTML格式
      application/xml XML数据格式
      application/atom+xml Atom XML聚合格式
      application/json JSON数据格式
      application/pdf PDF格式
      application/msword Word文档格式
      application/octet-stream 二进制数据流
      application/x-www-form-urlencoded
      中默认的encType，form表单数据被编码为key/value格式发送到服务器（表单默认的提交数据的格式

    • 另外一种常见的媒体格式是上传文件之时使用的：

      格式 说明
      multipart/form-data 表单进行文件上传格式

  • 添加请求头中的content-disposition : "inile;filename='${filename}'，其中inline是预览图片，attachment是下载附件

    // 设置强制下载不打开  
response.setContentType("application/force-download");
// ISO-8859-1 ==> UTF-8 进行编码转换
response.addHeader("Content-Disposition", "attachment;fileName=" + new String(file.getName().getBytes("UTF-8"),"iso-8859-1"));

//设置预览图片  
response.setContentType("image/png");
// ISO-8859-1 ==> UTF-8 进行编码转换
response.addHeader("Content-Disposition", "inline;fileName=" + new String(file.getName().getBytes("UTF-8"),"iso-8859-1"));
